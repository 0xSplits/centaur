🧭

The one-paragraph version. The Chat SDK already ships a first-class Discord adapter with the same interface as the Slack adapter in use today. The Rust api-rs control plane is platform-agnostic, so a discord:… thread key flows through unchanged. We clone slackbotv2 → discordbot, extract the drift-dangerous session logic into a shared package, swap the adapter, and add a single long-lived Gateway connection. The only hard, novel risk: Discord delivers normal messages over a Gateway WebSocket (not HTTP webhooks), which can't be proxied and must run in a long-lived pod.

Key Decisions

Four from the brainstorm, three resolved during planning. Every one is locked.

Structure · Brainstorm

🧬 Clone, don't generalize

New services/discordbot/; the Slack path stays untouched. A shared platform-parametrized service was rejected — it would couple the Slack runtime to Discord changes.

Ingress · Brainstorm

🔌 Persistent Gateway listener

Full parity needs the Gateway WebSocket (with the Message Content Intent). HTTP-only would give slash commands but no @-mention chat or thread follow-ups.

Behavior · Amended 06-01

🧵 Native Discord threading

Channel mention creates a public thread and answers inside it; in-thread mention replies to the message. Revives the Slack title as the Discord thread name. Status header → typing indicator.

Scope · Brainstorm

🚢 Deployment in scope

Dockerfile, Helm, CI, and Discord app setup are all planned. Built fresh off v1 slackbot — slackbotv2 has no deploy wiring yet either.

Architecture · Decided 06-01

📦 Extract @centaur/chat-session-bridge

Share the zero-platform SSE/Codex logic instead of forking it. Runtimes stay isolated; slackbotv2 switches to the package in the same change.

Concurrency · Decided 06-01

🔒 Keep onLockConflict: 'force'

Validated by a concurrent same-thread dedup test. The per-thread activeExecution guard already serializes execution.

Security · Decided 06-01

🛡️ Document + defer api-rs auth

The fail-closed allowlist + NetworkPolicy is the MVP boundary. api-rs bearer auth is a separate platform-wide ticket, not a blocker.

🔑

☁️

Where the answer lands — threaded by default

Mention in a channel

🧵 → creates a thread

The bot opens a public thread from the mention message, names it from the message text (titleFromMessage, ≤100 chars), and streams the answer inside. Follow-ups in that thread append with no re-mention. Session key: discord:guild:channel:newThreadId.

Mention inside a thread

↩️ → replies to the message

Streams the answer in the same thread as a reply referencing the specific mention message. Session key: the existing discord:guild:channel:threadId.

🧪

Two pins this adds. (1) The thread-creation side effect must be deduped before startThread — a RESUME replay must not spawn a second thread. (2) Whether @chat-adapter/discord exposes thread creation + reply references is a Phase 0 spike item; the fallback is the raw discord.js client we already hold (message.startThread / channel.send({ reply })). Needs the Create Public Threads permission.

08

Correctness Invariants

Assumptions the clone leans on that the cloned source does not obviously provide. Pin each before coding.

1
Direct-mode dispatch is load-bearing

Verify Gateway MessageCreate actually routes through Chat into onNewMention — today only the HTTP webhook drives the handlers. Phase 0 success criterion #1.

2
Reconnect dedup is NOT free — and now guards a side effect

The forwardedMessageIds guard re-executes a RESUME-replayed first mention. Add a persisted seen-message.id check at the gateway.ts boundary, running before startThread so a replay can't spawn a second thread. Design it in, don't defer to the test.

3
onLockConflict: 'force' semantics flip

One long-lived process means contenders are concurrent async tasks, not separate invocations. Kept, but the dedup test must fire two same-thread events concurrently.

4
No in-flight registry exists for drain

waitUntil pulls off the Hono request context — absent off the Gateway path. Define it as "push promise into a drain set"; SIGTERM drains → client.destroy() → close pool → exit.

5
Thread.allMessages may be empty on Discord

collectInitialContext backfills via it (Slack: conversations.replies). If unimplemented, the bot silently loses pre-mention context. Probe in Phase 0.

6
author.isMe/isBot must be populated

Or the bot replies to itself → infinite loop. Cheap Phase-0 check; catastrophic if wrong.

7
Typing keepalive needs error containment

The ~8s re-fire loop must swallow startTyping errors and clear its interval in finally, or a wedged stream leaks a loop / crashes Bun on an unhandled rejection.

Security Model

api-rs has no ingress auth — so the bot's allowlist is the primary authorization control, not defense-in-depth.

🔴 C1 — Deny DMs

A DM has guildId="@me" and reaches the tool-capable agent like any channel. Don't request DM intents; the allowlist rejects @me. One line, no DM-allowlist subsystem.

🔴 C2 — Fail-CLOSED guild allowlist

Empty DISCORDBOT_GUILD_ALLOWLIST ⇒ bot inert + loud warning. Cloning Slack's allow-by-default is unsafe — an invite link can place the bot in arbitrary servers.

🟠 H1 — Dedicated DISCORDBOT_API_KEY

Don't reuse the Slack key — session-api.ts reads it directly, so a metadata-only clone silently shares it. One env var.

🟠 H3 — Direct mode only; never log secrets

Assert webhookUrl unset (forwarding mode sends the bot token as a header); scrub the token from logs and error reflections.

📋

Hardening backlog — out of MVP scope (applies to Slack too): api-rs ingress bearer auth · per-user/per-guild rate limits · attachment size/MIME caps · raw-field minimization before persistence · channel-level allowlist granularity. Tracked as platform-wide tickets, not parity-blocking.
